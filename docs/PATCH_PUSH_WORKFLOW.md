# Patch Push Workflow — pushing local edits to canonical (v2.0.0)

> Single source of truth: `caudellhenry/foundry-pipeline`. This doc explains how local edits get pushed back to the canonical repo.

---

## Why this exists

When you install foundry-pipeline into your machine, your AI coding agent may edit the installed files (e.g., tweak a skill description, patch a hook script, add a custom template). Those edits are valuable — they often fix bugs or improve UX for your workflow.

The patch-push workflow gives you a one-command path to send those edits upstream as a PR to `caudellhenry/foundry-pipeline`, so the whole community benefits.

---

## Detection

`packages/core/scripts/foundry-self-update.sh` runs at SessionStart (every harness with hook support), pre-phase-transition, and on `/foundry:patch-check`. It supports two detection modes:

### Mode A — git-aware (when install dir is a git checkout)

```bash
LOCAL_HEAD=$(git -C "$INSTALL_DIR" rev-parse HEAD)
UPSTREAM_SHA=$(git ls-remote --tags "$UPSTREAM" "v$VERSION" | awk '{print $1}')
[[ "$LOCAL_HEAD" != "$UPSTREAM_SHA" ]] && emit_prompt
```

### Mode B — file-checksum (when install dir is not a git checkout)

```bash
LOCAL_SHA=$(sha256sum "$INSTALL_DIR"/skills/*/SKILL.md | sha256sum)
CANONICAL_SHA=$(curl -sL "$UPSTREAM_RAW/v$VERSION/.foundry-version-manifest.json" \
                | jq -r '.files."skills/foundry-ship/SKILL.md"')
[[ "$LOCAL_SHA" != "$CANONICAL_SHA" ]] && emit_prompt
```

The `.foundry-version-manifest.json` ships in every published tag (generated by `foundry-monorepo-build.sh` at build time).

## The prompt

When local diverges, you see (via SessionStart hook `additionalContext` or `/foundry:patch-check`):

```
⚠️  foundry v2.0.0 installed locally differs from canonical v2.0.0.

    3 files modified (skills/foundry-ship/SKILL.md, scripts/foundry-loop.sh, ...).
    1 unpushed commit ahead.

    Commands:
      /foundry:patch-diff    Show the diff vs canonical
      /foundry:patch-push    Push local changes to caudellhenry/foundry-pipeline
      /foundry:patch-reset   Discard local changes, reinstall canonical v2.0.0
      /foundry:patch-skip    Ignore this divergence (default 30 days)
```

## Commands

### `/foundry:patch-check`

Re-runs detection. Useful if you want a fresh prompt without waiting for SessionStart.

### `/foundry:patch-diff`

Shows the full diff between local and canonical:

```
diff --git a/packages/core/skills/foundry-ship/SKILL.md b/packages/core/skills/foundry-ship/SKILL.md
index 1234567..89abcde 100644
--- a/packages/core/skills/foundry-ship/SKILL.md
+++ b/packages/core/skills/foundry-ship/SKILL.md
@@ -12,6 +12,7 @@ description: |
+  Hints at PR template — caller should fill in: target version, breaking changes.
```

### `/foundry:patch-push`

The main workflow. Walks you through:

1. **Fork detection** — if you don't have a fork of `caudellhenry/foundry-pipeline`, the script prompts: "Fork caudellhenry/foundry-pipeline to your account? [y/N]". If `y`, it uses `gh repo fork` and clones your fork.

2. **Branch creation** — checks out `patch/<your-gh-user>/<YYYY-MM-DD>-<short-desc>` on your fork.

3. **Diff application** — applies your local changes on top of `v$VERSION`:
   ```bash
   git checkout v$VERSION
   git checkout -b patch/<user>/<date>-<short-desc>
   # Apply local diff:
   git --git-dir="$INSTALL_DIR/.git" --work-tree="$INSTALL_DIR" diff > /tmp/local.patch
   git apply /tmp/local.patch
   ```

4. **Eval gate** — runs `bash evals/run.sh --release-check`. If any release-gating scenario fails, the push is aborted with a clear error.

5. **PR creation** — opens a PR against `caudellhenry/foundry-pipeline:main` with auto-filled template:

   ```markdown
   ## Local patch

   - **Source harness**: <claude-code | zcode | …>
   - **Source version**: v2.0.0
   - **Files touched**: 3 (skills/foundry-ship/SKILL.md, scripts/foundry-loop.sh, …)
   - **Pass^k**: PASS (8/8 scenarios)
   - **Author**: @<your-gh-user>

   ## Diff summary

   <output of `git diff --stat v2.0.0..HEAD`>

   ## Checklist

   - [ ] I have read [CONTRIBUTING.md](../blob/main/CONTRIBUTING.md)
   - [ ] I have run `bash scripts/foundry-self-test.sh` locally
   - [ ] My changes follow the conventional-commits format
   ```

6. **Confirmation** — the script prints the PR URL and waits for your confirmation that you want to keep the local branch alive (in case you want to iterate).

### `/foundry:patch-reset`

Discards all local changes and re-installs canonical `v$VERSION`:

```bash
# Backs up local changes to ~/.foundry/patch-reset-backups/<timestamp>/
# Then re-clones or re-extracts from canonical tag
```

**Destructive** — this is what you run when the local edits were experiments you no longer want.

### `/foundry:patch-skip [days=30]`

Snoozes the divergence alert for N days. Writes a marker file:

```
~/.foundry/patch-skip-until: 2026-08-04
```

Until that date, `/foundry:patch-check` exits 0 silently.

## What gets pushed (and what doesn't)

**Pushed**:
- Skill files (`SKILL.md`)
- Script changes (`scripts/*.sh`)
- Hook changes (`hooks/*.sh`)
- Template changes (`templates/*.md`)
- Eval scenario additions/changes

**Not pushed**:
- Per-project state files (`.foundry/state.md` — never ship your project state)
- Personal config (`.foundry/patch-skip-until`, etc.)
- Custom skills that aren't in canonical (those are local-only; consider `/foundry:init` to add them to your `personal:` skills registry)

---

## Conflict resolution

If your local edits conflict with newer canonical changes (someone else merged a patch first), `/foundry:patch-push`:

1. Tries `git rebase v$VERSION` automatically.
2. If rebase conflicts, prints the conflicting files and exits with code 1 + instructions to resolve manually.
3. Does NOT force-push to your fork. Uses `--force-with-lease` only on the patch branch.
